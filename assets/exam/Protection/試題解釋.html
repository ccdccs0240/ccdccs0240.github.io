113-2, 1

這個問題的正確答案是 (C)。

錯誤的敘述解釋

(C) 中間人攻擊（Man in the middle attack）： 題目中對中間人攻擊的描述混淆了 中間人攻擊 (MITM) 和 分散式阻斷服務攻擊 (DDoS) 的概念。

  • 中間人攻擊 (MITM) 的核心是攻擊者秘密地攔截並可能修改兩個通訊方之間的對話，讓他們以為是直接互相通訊。它的重點在於「竊聽、篡改」通訊內容。
  • 分散式阻斷服務攻擊 (DDoS) 才是使用多台受感染電腦（殭屍網路）向目標發起大量請求，目的是使目標服務癱瘓或不可用。

因此，將「使用多台電腦發起大量請求」歸因於中間人攻擊是錯誤的。

正確的敘述解釋

  • (A) SQL 注入式攻擊（SQL Injection）： 這個敘述是正確的。攻擊者利用應用程式對使用者輸入未經適當驗證的漏洞，將惡意的 SQL 語句插入到資料庫查詢中，從而繞過安全措施，執行非預期的資料庫操作，如查詢、修改或刪除資料。
  • (B) 跨站腳本攻擊（Cross-site Scripting，XSS）： 這個敘述是正確的。XSS 攻擊是指攻擊者將惡意腳本（通常是 JavaScript）注入到一個合法且受信任的網站中。當其他使用者瀏覽該網站時，他們的瀏覽器會執行這些惡意腳本，導致會話劫持、敏感資訊竊取（如 Cookie）、或將使用者重新導向到惡意網站等危害。
  • (D) 釣魚攻擊（Phishing）： 這個敘述是正確的。釣魚攻擊是一種社會工程學（Social Engineering）手法，攻擊者偽裝成可信任的實體（例如銀行、政府機構、知名公司等），透過電子郵件、簡訊、電話或偽造網站等方式，誘騙受害者洩露個人敏感資訊，如登入憑證、信用卡號、身份證號碼等。

113-2, 2

這個問題的正確答案是 (A)。

正確的敘述解釋

(A) APT 常用技術手法，如：Privilege Escalation 提權、Lateral Movement 橫向移動 這個敘述是正確的。APT 攻擊的特點是長期潛伏、多階段、針對性強。攻擊者一旦成功滲透進入目標網路，會利用各種技術手段來擴大其控制範圍並維持存取權限：

  • 提權 (Privilege Escalation)： 攻擊者會嘗試獲取更高的系統權限，例如從普通使用者權限提升到管理員權限，以便執行更多操作。
  • 橫向移動 (Lateral Movement)： 攻擊者會在目標網路內部從一台被攻陷的電腦移動到另一台電腦，逐步擴大其在網路中的立足點，尋找並存取有價值的資料或系統。

錯誤的敘述解釋

  • (B) 常見於一般腳本小子（Script Kiddie） 這個敘述是錯誤的。APT 攻擊通常由國家級駭客組織、專業犯罪集團或具有高度技術能力的團體發起。這些攻擊需要大量的資源、時間、專業知識和耐心。而「腳本小子」通常是指技術能力較低，主要使用現成工具或腳本進行簡單、非針對性攻擊的人。APT 攻擊的複雜性和精密性遠超腳本小子的能力範圍。
  • (C) 非針對特定對象之入侵行為 這個敘述是錯誤的。APT 的核心特徵就是「針對性 (Targeted)」。攻擊者會精心選擇特定的目標，例如政府機構、關鍵基礎設施、大型企業或擁有高價值智慧財產權的組織。他們會花費大量時間進行情報收集和偵察，為目標量身定制攻擊策略。
  • (D) 利用零日漏洞、客製化的惡意軟體，缺點是不具備匿名性與持久性 這個敘述是錯誤的。
    • 利用零日漏洞和客製化惡意軟體： 這部分是正確的。APT 攻擊確實經常利用未公開的零日漏洞 (Zero-day Vulnerabilities) 和專為目標設計的客製化惡意軟體，以提高攻擊的成功率和隱蔽性。
    • 缺點是不具備匿名性與持久性： 這部分是錯誤的。APT 的另一個核心特徵是「持久性 (Persistent)」。攻擊者會想方設法在目標網路中長期潛伏，維持其存取權限，即使被發現部分入侵，也能迅速恢復。同時，他們也會採取多種措施來保持「匿名性」，避免被追溯到真實身份或來源。因此，匿名性和持久性恰恰是 APT 攻擊者極力追求的目標，而非缺點。

    解釋： 社交工程（特別是釣魚攻擊）是紅隊演練中非常重要的環節，因為「人」往往是防禦鏈中最薄弱的一環。攻擊者會針對特定的員工或部門發送高度客製化的釣魚郵件，目的是誘使員工執行某些操作（例如點擊惡意連結、下載惡意附件或洩露憑證），從而取得員工電腦的控制權或存取其敏感資料，進一步利用這些資料來推進攻擊。

    解釋： 這屬於實體入侵或無線網路滲透的範疇，也是紅隊演練會考慮的攻擊路徑之一。紅隊人員會模擬攻擊者實際到達目標地點，嘗試透過無線網路的弱點（如弱密碼、配置錯誤）來入侵企業網路。這能測試企業在實體安全和無線網路安全方面的防禦能力，以及內部網路隔離措施的有效性。

不屬於典型紅隊演練的攻擊手法解釋

  • (D) 註冊成為企業的供應商，並且利用供應商專用網站造假資料、竄改商品金額，驗證攻擊者可以用更低的價格買進商品

    解釋： 雖然這是一種供應鏈層面的安全漏洞或業務邏輯漏洞，並可能導致欺詐行為，但它更偏向於業務流程測試 (Business Process Testing) 或 詐欺風險評估 (Fraud Risk Assessment)，而非傳統定義的「紅隊演練」主要關注的點。紅隊演練的核心目標是模擬真實世界的網路攻擊，評估企業在「偵測、防禦和應對」入侵行為的能力，通常涉及網路滲透、資料竊取、系統控制等。而竄改商品金額等行為，雖然是資訊安全問題，但其直接目的和技術路徑與典型的網路入侵模擬有所區別。

NIST SP800-30 對「弱點（Vulnerability）」的定義： 「系統安全程序、設計、裝置、或內部控制的一個瑕疵或缺陷，若被利用後，會破壞安全性或違背系統安全政策。」

  • (A) 微軟發布之 Windows 作業系統漏洞

    判斷： ✔ 正確描述了弱點。

  • (B) 系統管理人員依程序於每季末將離職人員之帳號停用

    分析： 雖然這是按照程序執行，但「每季末」才停用離職人員帳號，這表示在一個季度內，離職員工的帳號可能仍然活躍。這是一個「系統安全程序或內部控制」上的「缺陷」（即延遲性或不即時性）。若此缺陷被利用（例如，離職員工在帳號被停用前仍能存取系統），將會違背系統安全政策。
    判斷： ✔ 正確描述了弱點（即程序本身存在的缺陷）。

  • (C) IoT 設備無法變更預設密碼及帳號

    分析： 「無法變更預設密碼及帳號」是「系統設計或裝置」上的「瑕疵或缺陷」。如果這些預設密碼和帳號被攻擊者利用，將會破壞安全性。這完全符合弱點的定義。
    判斷： ✔ 正確描述了弱點。

  • (D) 組織內部人員違反資安規定，私自攜帶外接式儲存設備並連接至組織內部重要伺服器使用，進而導致檔案伺服器遭受勒索病毒感染，重要資料遭受加密無法使用

    分析： 這個敘述描述的是一個安全事件的發生過程和結果，即「內部人員違規」->「連接外部設備」->「導致勒索病毒感染」->「資料被加密」。雖然這個事件的發生根源於某個「弱點」（例如，資安規定執行不力、缺乏有效的技術控制來阻擋外接設備、或伺服器本身的漏洞），但敘述本身並非在描述「一個瑕疵或缺陷」本身，而是在描述「弱點被利用後的破壞行為和結果」。弱點是導致事件發生的潛在原因，而不是事件的完整過程或其造成的後果。
    判斷： ✘ 錯誤描述了弱點。它描述的是弱點被利用後的安全事件與其影響，而非弱點本身。

    分析： CVSS 的數值分數確實可以轉換為「低、中、高、嚴重」等定性等級。這種轉換有助於組織更直觀地理解弱點的影響，並有效地排列處理弱點的優先順序，從而優化弱點管理流程。
    判斷： ✔ 正確。

    分析： 這是 CVSS 評分系統的標準結構。

    • 基本度量組 (Base Metrics)：描述弱點的固有特性，如攻擊向量、影響範圍等。

針對「機關對外服務網站無法正常提供服務，經分析 DNS 遭受混合式 DDoS 攻擊，下列建議之 DDoS 防護措施何者最『不』合適？(複選)」這個問題，以下是我的詳細分析和判斷：

    分析： 網路流量清洗服務（或稱 DDoS 緩解服務）是專為應對大規模 DDoS 攻擊而設計的。它們能將受攻擊目標的流量引導至專用的清洗中心，在那裡識別並過濾掉惡意流量，只將合法流量轉發給目標。對於混合式 DDoS 攻擊，特別是高容量攻擊，這是目前最有效且推薦的解決方案之一。
    判斷： ✔ 非常合適。

  • (B) 可透過限制 DNS 查詢次數，來緩解 DDoS 攻擊

    分析： 限制 DNS 查詢次數（Rate Limiting）是針對 DNS 伺服器應用層攻擊的有效防禦措施。它可以防止單一來源或有限的惡意來源透過發送大量查詢來耗盡 DNS 伺服器的處理資源。然而，面對混合式 DDoS 攻擊，尤其是當攻擊流量來自高度分散的殭屍網路，且每個惡意來源的查詢速率都保持在限制以下時，僅靠此措施可能無法完全阻止聚合流量對頻寬或伺服器整體資源的消耗。雖然這是一項重要的安全配置，但在大規模分佈式攻擊面前，其獨立的緩解能力有限。
    判斷： ✘ 不夠合適（因其在應對高度分散和大規模攻擊時的局限性）。

    分析： 這項敘述的前提「如果網路頻寬無須考量」從根本上與 DDoS 攻擊的定義相悖。DDoS 攻擊的關鍵目標就是透過巨量流量耗盡目標的網路頻寬，使其無法提供正常服務。如果頻寬無限，那麼流量型的 DDoS 攻擊本身就無法達成其主要目的。在實際情況中，頻寬總是一個重要的考量因素。雖然過濾機制（例如防火牆規則、ACL）是 DDoS 防護的組成部分，但在缺乏足夠頻寬的情況下，這些本地過濾措施在流量抵達網路設備並被處理之前，可能就已經導致上游鏈路飽和。因此，將此措施建立在一個不切實際的假設上，使其成為不適用於實際 DDoS 情境的建議。
    判斷： ✘ 最不合適（因其前提條件與 DDoS 攻擊的本質相矛盾）。

    判斷： ✔ 非常合適。

PTES 滲透測試執行標準 (Penetration Testing Execution Standard) 流程正確順序：

  • 前期溝通 (Pre-engagement Interactions)
    建立測試範圍、目標、合約與法規範定，確認滲透測試的細節與雙方共識。
  • 情報收集 (Intelligence Gathering)
    蒐集目標環境的公開資訊，例如 DNS、IP、子網路、作業系統、服務等，進行基礎架構與弱點盤點。

    分析蒐集到的資料，模擬攻擊者可能採取的攻擊路徑、目標與資產評估。
  • 漏洞分析 (Vulnerability Analysis)
    掃描、分析並確認目標系統的已知與未知漏洞。
  • 利用 (Exploitation)
    利用已發現的漏洞進行滲透、提升權限、進一步存取目標系統與網路。
  • 後期利用 (Post Exploitation)
    進行持續性訪問、數據竊取、後門設置，並評估攻擊對業務的影響。
  • 報告 (Reporting)
    將測試過程、發現的漏洞、影響與修復建議完整呈現，提供給組織進行後續防護措施。

    錯誤：滲透測試可能會模擬攻擊，對資料庫執行大量測試操作，這可能會影響資料庫的穩定性。雖然可以檢測安全性，但並不適合在此情境下作為「初步的安全驗證」。

  • (D) 提交一份主機弱點檢測報告

    錯誤：主機弱點檢測（如 Windows/Linux）主要檢查 OS 層級的漏洞，不涉及資料庫本身的安全設定。

為何選擇資料庫安全健檢？

資料庫安全健檢 (Database Security Assessment) 是對資料庫內部進行安全評估，如：

  • 帳號密碼政策：密碼複雜度、帳號鎖定機制。
  • 存取權限：檢視哪些帳號有不必要的高權限。
  • 日誌與審計：資料庫的審計功能是否開啟、日誌是否被妥善保存。
  • 加密與備份：資料傳輸與儲存的加密措施。
  • 補丁更新：資料庫版本與安全性更新是否及時。

各選項分析：

    發現惡意攻擊的殘餘指令碼，高度疑似存在安全漏洞尚未修補，攻擊者才能入侵並留下痕跡。立即進行漏洞修補，更新作業系統、應用程式、及防護軟體，是標準應對措施。

  • ✅ (B) 需要對系統進行更深入的檢視

    單一滲透測試的發現只是冰山一角，需要進一步檢查整體系統：

    • 是否有後門或Rootkit尚未發現？
    • 系統有無異常帳號、異常流量或可疑的排程任務？
    • 需要進行全面的惡意碼掃描與記錄分析，以確認系統未受持續控制。

  • ❌ (C) 需要增強員工對釣魚攻擊和社交工程的警覺性

    此情境是「發現攻擊殘餘的指令碼」，重點在於系統被攻擊，而非員工因釣魚攻擊導致入侵。此選項雖然是長期資安策略之一，但此情境下並非最直接且必要的措施。

  • ✅ (D) 應該檢查並更新防火牆和入侵檢測系統的配置

    外部攻擊成功入侵，可能表示現有防火牆規則或入侵檢測系統（IDS/IPS）存在缺陷或配置不足。檢查並更新防禦設備的規則、簽章及配置，可防止未來類似攻擊再次成功。
