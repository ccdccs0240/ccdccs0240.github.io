解析：

MITRE ATT&CK 框架將攻擊者的行為分為多個戰術（Tactics）與技術（Techniques）。

橫向移動（Lateral Movement） 是攻擊者在已滲透的內部網路中橫向擴展到其他系統的行為，包含使用合法的憑證或工具來控制更多系統。

正確選項說明：

  • (A) 遠端服務破解（Exploitation of Remote Services）：攻擊者可能會利用目標主機上的漏洞，透過網路服務進行橫向移動。
  • (B) 內部魚叉釣魚（Internal Spearphishing）：攻擊者在已滲透的網路內部進行魚叉式網路釣魚，進一步入侵更多主機。
  • (C) 橫移工具傳輸（Lateral Tool Transfer）：將攻擊工具從已入侵的系統傳輸到其他目標系統以協助橫向移動。

錯誤選項說明：

  • (D) 本地資料取得（Data from Local System）：此技術屬於 資料竊取（Collection）戰術，與橫向移動無關，故為錯誤選項。

(C)

解析：

DDoS 攻擊 是分散式阻斷服務攻擊，主要目的是癱瘓目標系統或網路，使其無法提供正常服務。攻擊方式常見的類型包括：

  • 消耗目標的網路頻寬（如 UDP Flood、ICMP Flood）
  • 耗盡目標伺服器的資源（如 SYN Flood、HTTP Flood）

然而，DDoS 攻擊的重點在於「癱瘓服務」，而非擷取網路上的傳輸封包資訊。

封包攔截 與擷取通常屬於中間人攻擊（MITM），並不屬於 DDoS 攻擊的類型。

選項說明：

  • (A) 正確：

    當時唐鳳部長的確提到使用 Web3 技術 建立「不對稱防禦架構」，如「去中心化網頁鏡像」，以減少攻擊對單一目標的影響。

  • (B) 正確：

    流量清洗 是一種常見的 DDoS 緩解措施，透過篩選正常與異常流量，避免惡意流量癱瘓系統。但這確實會帶來大量的資源與成本。

  • (C) 錯誤：

    DDoS 攻擊的核心是「癱瘓」而非「擷取資訊」，此選項描述錯誤，因此為正確答案。

(C)

解析：

當發現不明的異常流量時，資安事件處理的首要步驟是分析與確認威脅來源，而非立即採取極端或非必要行動。依據資安事件應變流程（Incident Response Process），應先進行：

  • 偵測與識別：確定異常流量是否為攻擊，並收集相關證據（如流量來源 IP、異常行為類型等）。
  • 分析與評估：進一步進行威脅與風險分析，確保掌握攻擊範圍與影響程度，再擬定應對策略。

選項說明：

  • (A) 關閉所有的內部網路 ❌

    立即關閉內部網路可能造成業務中斷，並非初期必要的反應，除非確認攻擊嚴重且無法控制。

  • (B) 通知所有員工有潛在的威脅 ❌

    在威脅尚未確認時，貿然通知全體員工可能引發不必要的恐慌，應先確認威脅，再有條理地通知相關人員。

  • (C) 進行詳細的威脅與風險分析 ✔️

    正確！先詳細分析異常流量的來源、類型、影響範圍等，才能進一步制定有效的應對措施。

  • (D) 立即去法院按鈴申告 ❌

    法律行動通常在確認攻擊並蒐集足夠證據後才進行，初期應優先處理技術層面的分析與防護。

(B) 事件回應團隊

解析：

在確認並瞭解詳細的威脅情況後，第一步應該立即與事件回應團隊（Incident Response Team，IRT）合作，執行資安事件的應變與處理。

為什麼選 (B) 事件回應團隊？

事件回應團隊 是專門負責資安事件處理的專業團隊，負責：

  • 針對異常或攻擊事件做出反應。
  • 執行必要的遏止、清除與恢復行動。
  • 蒐集證據、追蹤攻擊來源，並確保業務影響最小化。

其他選項分析：

  • (A) 市場行銷團隊 ❌

    行銷團隊與資安事件處理無直接關聯，除非需要在後期進行對外溝通或公告，但絕非第一步的合作對象。

  • (B) 事件回應團隊 ✔️

    正確！事件回應團隊是資安事件應變的核心，會在確認威脅後立即展開防禦、遏止與後續復原工作。

  • (C) 威脅分析團隊 ❌

    威脅分析團隊主要負責事前的威脅情報收集與分析，而非資安事件爆發後的第一線處理團隊。

    （可能在後期提供支持，但不是「第一步」）

  • (D) 行銷風險評估團隊 ❌

    與行銷相關的風險評估團隊並非專門處理資安事件的團隊，這與資安事件應變流程無關。

(C) Delivery

解析：

Cyber Kill Chain 是由 Lockheed Martin 提出的攻擊鏈模型，描述了一次網路攻擊的完整過程。

若攻擊來源是透過郵件附件進行攻擊，該行為對應的階段是：

Delivery（投遞階段）

  • Delivery 是指攻擊者將惡意載荷（如惡意附件、連結等）投遞到受害者環境中的過程。
  • 郵件附件 是典型的投遞方法，攻擊者透過釣魚郵件將惡意程式送到受害者手中。

其他選項說明：

  • (A) Weaponization（武器化） ❌

    此階段指的是攻擊者製作惡意軟體、漏洞利用工具或載荷，準備攻擊武器，並非投遞行為本身。

  • (B) Installation（安裝） ❌

    指的是惡意程式在受害者系統中成功執行並安裝的過程，這是發送郵件附件之後的階段。

  • (C) Delivery（投遞） ✔️

    正確！郵件附件是攻擊的「投遞工具」，此行為屬於 Delivery 階段。

  • (D) Exploitation（利用） ❌

    此階段指的是攻擊者利用系統漏洞來執行惡意程式或提升權限，已超過「附件投遞」這個初始行為。

📚 Cyber Kill Chain 七個階段（IPAS考試常考）：

  • Reconnaissance：偵察與情報收集
  • Weaponization：武器化（製作攻擊載荷）
  • Delivery：投遞（發送釣魚郵件、惡意連結等）
  • Exploitation：利用（利用漏洞執行惡意程式）
  • Installation：安裝（在受害系統植入後門）
  • Command & Control (C2)：指揮與控制（建立通訊通道）
  • Actions on Objectives：目標行動（進行竊取資料、破壞等行為）

（A）

關於雜湊函數（Hash Function），我們來逐一分析選項：

  • ✅ (B) 常見的雜湊函數有 MD5、SHA-1 及 SHA-2 等

    → 正確，這些都是常見的雜湊演算法，其中 MD5 和 SHA-1 已被視為不安全，SHA-2（如 SHA-256、SHA-512）仍然被廣泛使用。

  • ✅ (C) 輸入的原始字串只差一個字元，透過雜湊函數得到的結果會差異很大

    → 正確，這是「雪崩效應（Avalanche Effect）」的特性，即即使輸入值只變動一點點，輸出的雜湊值也會大幅改變。

  • ✅ (D) 雜湊函數具有不可逆的特性，無法利用雜湊值計算取得原始資料

    → 正確，雜湊函數是一種「單向函數」，不能從雜湊值逆推回原始數據，因此適合用來儲存密碼。

  • ❌ (A) 雜湊原理是限制輸入長度為 8 字元以上並透過雜湊函數進行計算，計算後會得到一樣長度的密文

    → 錯誤，雜湊函數對輸入長度沒有特定限制，它可以接受任何長度的輸入，且產生固定長度的輸出。例如：

    • SHA-256 會輸出 256 位元（32 字節） 的雜湊值，無論輸入長度多少。
    • SHA-512 則會輸出 512 位元（64 字節） 的雜湊值。

    此外，雜湊（Hash）並不等同於加密（Encryption），它不屬於「密文」，而是「摘要」。

(A)

SQL Injection: SQL 注入攻擊

解析：

根據 OWASP Top 10（開放式 Web 應用程式安全計畫）所列出的最常見 Web 應用程式安全弱點，SQL Injection（SQL 注入攻擊） 是最具危害性的攻擊方式之一。

  • SQL Injection (SQLi)：攻擊者透過惡意 SQL 語句插入到應用程式的查詢中，進而竊取、篡改或刪除資料庫中的數據，甚至獲取系統控制權限。
  • 這類攻擊通常發生在未經過適當 輸入驗證與 SQL 語法轉譯 的 Web 應用程式中。
  • 由於許多應用程式都與資料庫交互，SQLi 是最常見且影響嚴重的 Web 弱點之一。

其他選項分析：

(B) DDoS攻擊：屬於網路層攻擊，主要透過大量流量癱瘓伺服器，不屬於 Web 應用程式內部的安全弱點。

(C) LFI（本地端檔案包含漏洞）：雖然常見，但比起 SQL Injection，影響範圍相對較小。

(D) CSRF（跨站請求偽造）：確實是 Web 應用程式的弱點，但影響範圍通常受限於受害者的權限範圍，相較於 SQL Injection，攻擊影響較小。

結論：

Web 應用程式最常見且影響最嚴重的安全弱點是 SQL Injection（SQL 注入攻擊），所以選擇 (A)。

(B) 無法透過觀察驅動程式載入（Driver Load）活動來偵測此技術

解析：

MITRE ATT&CK 中的 輸入擷取（Input Capture，T1056） 技術主要用於竊取使用者輸入的資訊，例如鍵盤輸入、滑鼠點擊、剪貼簿內容等。這類技術經常被惡意軟體（如鍵盤側錄器）利用來竊取憑證或敏感數據。

選項分析：

  • ✅ (A) 無法使用預防控制（Preventive Controls）輕易緩解此技術 → 正確

    輸入擷取技術通常是惡意程式透過 Hook API、記錄鍵盤事件等方式運行，難以單純依靠預防控制（如防火牆、傳統防毒軟體）來阻止。

    需要 行為分析、端點防護（EDR）、應用白名單 等進階技術來減少風險。

  • ❌ (B) 無法透過觀察驅動程式載入（Driver Load）活動來偵測此技術 → 錯誤（應該可以偵測）

    許多鍵盤側錄技術（如 API Hooking、內核級 Keylogger）需要安裝額外的 驅動程式（Driver） 來攔截輸入事件，因此可以透過 監控驅動程式載入活動 來偵測。

    例如：監控 kbdclass.sys、mouclass.sys 這類鍵盤與滑鼠驅動程式是否被惡意替換或掛鉤。

  • ✅ (C) 可觀察系統註冊機碼（Windows Registry）活動來偵測此技術 → 正確

    某些 Keylogger 會 修改註冊表鍵值（如 HKCU\Software\Microsoft\Windows\CurrentVersion\Run）來確保惡意程式開機自啟動。

    可透過監控 Run、RunOnce、Startup 這些註冊機碼來偵測異常活動。

  • ✅ (D) 可觀察程序建立（Process Creation）活動與元資料（Process Metadata）來偵測此技術 → 正確

    Keylogger 可能會產生異常的 進程行為，例如開啟鍵盤事件的 API（如 GetAsyncKeyState、SetWindowsHookEx）。

    可以透過 EDR 工具 監控進程行為，發現可疑的鍵盤側錄行為。

(C) 電子郵件附件或內文 URL 連結

解析：

Cyber Kill Chain 模型是由 Lockheed Martin 提出的攻擊階段框架，其中 "Delivery"（傳遞） 階段代表攻擊者將惡意軟體或攻擊載荷（Payload）傳送給受害者的過程。

在這個階段，攻擊者會 使用電子郵件、惡意網站、社交工程或其他手段，試圖讓受害者下載或執行惡意軟體。

選項分析：

  • ✅ (C) 電子郵件附件或內文 URL 連結 → 正確

    這是 釣魚攻擊（Phishing） 的典型手法，攻擊者透過電子郵件夾帶 惡意附件 或 釣魚網站連結，誘騙受害者點擊並下載惡意程式。

    這符合 Delivery 階段，因為攻擊者正在「傳遞」惡意軟體到受害者端。

  • ❌ (A) 惡意軟體安裝 → 錯誤（屬於 Exploitation/Installation 階段）

    惡意軟體的「安裝」通常發生在 Exploitation（利用） 或 Installation（安裝） 階段，而非「Delivery」階段。

    Delivery 只是傳遞，受害者點擊後才會進入安裝過程。

  • ✅ (B) 網站水坑攻擊（Watering Hole Attack） → 部分正確，但不是最直接答案

    網站水坑攻擊是攻擊者入侵常被目標族群訪問的網站，並植入惡意代碼，讓訪客無意間感染惡意軟體。

    雖然這可以視為「Delivery」的一種方式，但更常見的「Delivery」方法還是 電子郵件附件與 URL，因此 (C) 更符合。

  • ❌ (D) 帳號密碼填充攻擊（Credential Stuffing） → 錯誤（屬於 Credential Access 階段）

    帳密填充攻擊是攻擊者利用 洩漏的帳號密碼組合 來測試是否能登入其他服務，這屬於 Credential Access（憑證存取） 階段，而非「Delivery」。

弱點管理（Vulnerability Management）是一個持續的流程，包含 識別、評估、修補、監控 弱點，以減少組織遭受攻擊的風險。

選項分析：

  • ✅ (A) 定期進行弱點掃描 → 必要措施

    使用 弱點掃描工具（如 Nessus、Qualys、OpenVAS） 來識別已知漏洞，例如：

    • 過時軟體
    • 缺少安全更新
    • 錯誤的設定（Misconfiguration）

    定期執行可以幫助 IT 團隊發現問題，並優先處理高風險漏洞。

  • ✅ (B) 使用最新的防毒軟體 → 輔助保護措施

    雖然防毒軟體無法 直接修補系統弱點，但它可以：

    • 偵測並攔截已知的惡意軟體
    • 防止攻擊者利用漏洞植入惡意程式（如勒索軟體、木馬）
    • 提供行為分析，檢測異常活動（部分 EDR/AV 具備此功能）

    防毒軟體與 入侵防禦系統（IPS）、端點防護（EDR） 搭配使用，能夠強化防禦。

  • ❌ (C) 仔細研讀所有的軟體使用條款 → 與弱點管理無直接關係

    研讀使用條款有助於了解軟體授權與隱私政策，但 不會影響漏洞識別與修補流程，因此 不是弱點管理的核心措施。

  • ✅ (D) 實施弱點修補管理流程 → 關鍵措施

    發現漏洞後，應該透過 修補管理流程（Patch Management） 來進行風險控管，包括：

    1. 評估弱點影響（如 CVSS 分數）
    2. 設定優先級（Critical > High > Medium）
    3. 測試補丁（避免影響系統運作）
    4. 部署修補（Windows Update、Linux Patch、應用程式修補）
    5. 監測與驗證（確保補丁成功應用）

結論：

正確答案為 (A)、(B)、(D)，因為這三項措施是 弱點管理的關鍵步驟，有助於減少組織風險。

📌 完整的弱點管理流程應包含：

1️⃣ 識別（Identify） → 弱點掃描（Vulnerability Scanning）

2️⃣ 評估（Assess） → 風險分析（CVE、CVSS）

3️⃣ 修補（Remediate） → 部署修補、配置調整

4️⃣ 驗證（Verify） → 重新掃描與持續監控

🔥 補充：為什麼 (B) 也是必要措施？

雖然 防毒軟體 無法直接修補漏洞，但它可以：

  • 檢測並攔截漏洞利用攻擊（如 Exploit Kits）
  • 防止惡意程式感染（如利用零時差漏洞的攻擊）
  • 透過 EDR 進行行為分析與即時防護

因此，雖然弱點管理的核心是 掃描與修補漏洞，但 最新的防毒軟體 仍然是重要的 補充防禦措施，所以 (B) 也是正確答案！

MITRE ATT&CK 瀏覽器會話劫持（T1539 - Browser Session Hijacking） 指的是攻擊者竊取有效的瀏覽器認證會話，藉此冒充受害者進行未授權存取，而不需要重新輸入密碼。這通常發生於：

  • Session Cookie 竊取（如 XSS、惡意軟體、MitM 攻擊）
  • 瀏覽器內存抓取（惡意程式從記憶體讀取會話資訊）
  • 憑證存取（透過 Infostealer 惡意軟體或瀏覽器漏洞）

選項分析：

  • ✅ (A) 可透過資安認知訓練（Awareness Training）來緩解此技術 → 正確

    許多瀏覽器會話劫持攻擊來自於 社交工程（Social Engineering），如：

    • 釣魚網站（Phishing）：騙取憑證或 Cookie
    • 誘導點擊惡意連結（例如惡意 JavaScript 竊取 Session）
    • 社交工程誘導禁用安全功能（如 2FA）

    因此，透過資安認知訓練，可以教育員工如何防範 釣魚攻擊、可疑網站、未授權瀏覽器擴充功能 等，提高安全意識，減少受害機率。

  • ✅ (B) 可分析登入會話（Logon Session）的產生狀態來偵測此技術 → 正確

    瀏覽器會話劫持後，攻擊者通常會 劫持現有的登入會話，不需要重新輸入密碼。

    偵測方式包括：

    • 異常的 Session ID 使用（同一帳戶在不同地點、裝置或 IP 登入）
    • 短時間內多次登入（特別是來自不同地區）
    • Session 沒有登入憑證，但執行敏感操作（如修改帳戶、匯款）

    可透過 SIEM（Security Information and Event Management） 監控登入記錄，發現異常行為。

  • ✅ (C) 可透過使用者帳戶管理（User Account Management）來緩解此技術 → 正確

    帳戶安全機制 可以降低會話劫持風險，包括：

    • 強制重新驗證（Reauthentication）（敏感操作前要求重新輸入密碼或 2FA）
    • 短會話時效（Session Timeout）（減少攻擊者利用時機）
    • 多因素驗證（MFA）（即使 Session 被盜，攻擊者仍需額外驗證）
    • 異常行為偵測與鎖定（Anomaly Detection）（如異地登入提醒）

    透過 適當的帳戶管理策略，可以大幅降低攻擊成功機率。

  • ❌ (D) 僅能透過分析程序存取（Process Access）與程序修改（Process Modification）狀態來偵測此技術 → 錯誤

    雖然攻擊者可能使用惡意軟體 讀取瀏覽器記憶體 或 修改進程 來竊取 Session（如 infostealer 惡意軟體），但這 不是唯一的偵測方式。

    其他偵測方式還包括：

    • 監控異常登入行為（如異常 Session ID 變更）
    • 分析網路流量（異常的 Cookie 傳輸或未授權裝置存取）
    • 監測瀏覽器異常活動（如瀏覽器進程訪問未預期的 API）

    因此，(D) 說 「僅能」透過程序存取與修改來偵測，是錯誤的。

結論：

正確答案： (A)、(B)、(C)

這三項措施都能幫助 偵測、預防、緩解瀏覽器會話劫持攻擊。

  • ✅ 啟用 HTTP Only & Secure Cookies（防止 JavaScript 竊取 Cookie）
  • ✅ 強制 HTTPS，防止中間人攻擊（MitM）
  • ✅ 實施多因素驗證（MFA）
  • ✅ 監控異常登入行為與 Session ID 變更
  • ✅ 定期清除 Cookie & 限制會話存活時間

🔹 人為層面：

  • ✅ 資安意識訓練（如防範釣魚攻擊）
  • ✅ 避免使用來路不明的瀏覽器擴充功能
  • ✅ 不在公共 Wi-Fi環境下登入敏感網站

(C) 確認並評估攻擊的範圍和影響

解析：

在 ISO 27001資訊安全管理體系（ISMS） 的框架下，當核心資訊系統受到攻擊時，應依照 事件處理流程（Incident Response Process） 來應對。

事件處理通常包含以下 六個步驟（SANS 事件處理模型）：

1️⃣ 準備（Preparation） → 事先規劃應變計畫

2️⃣ 識別（Identification） → 確認攻擊發生，並評估影響範圍 ✅

3️⃣ 遏制（Containment） → 防止擴散，暫時隔離受影響系統

4️⃣ 根除（Eradication） → 移除攻擊源與惡意軟體

5️⃣ 恢復（Recovery） → 修復系統、恢復正常運作

6️⃣ 經驗回顧（Lessons Learned） → 事後分析與改善

選項分析：

  • ✅ (C) 確認並評估攻擊的範圍和影響 → 正確

    這是事件處理的 第一步，需要確認：

    • 攻擊類型（如 DDoS、勒索軟體、SQL 注入）
    • 影響範圍（哪些系統受影響？哪些數據被存取？）
    • 攻擊持續時間（攻擊是否仍在進行？）
    • 攻擊來源（是內部人員、外部駭客，還是自動化攻擊？）

    若未先評估範圍，直接修復或通報，可能導致錯誤決策或影響取證。

  • ❌ (A) 修復受影響的系統 → 錯誤（時機不對）

    修復應該在 確認攻擊範圍、遏制攻擊 之後進行，而不是第一步。

    若直接修復，可能會：

    • 破壞取證證據（影響後續調查）
    • 忽略隱藏的後門（駭客可能仍有存取權限）
    • 錯誤評估影響範圍（攻擊可能影響更多系統）
  • ❌ (B) 報告給執法機關 → 錯誤（時機不對）

    報警前應先確認攻擊的範圍與性質，有些內部事件不一定需要報警。

    企業內部應先進行初步調查，判斷是否符合法規通報標準，如：

    • 是否涉及 GDPR 或個資法（影響個資才需要通報）
    • 是否影響 關鍵基礎設施（如政府機構、醫療、金融）
    • 是否涉及 刑事犯罪（如勒索軟體攻擊、金融詐騙）

    企業通常會先由 資安團隊 & 法律部門評估 是否報案，而不是第一時間報警。

  • ❌ (D) 通知所有受影響的客戶和使用者 → 錯誤（時機不對）

    除非確定影響客戶，否則不應過早發布通知，避免不必要的恐慌。

    需要 確認攻擊範圍後，依照企業 內部與法規要求 決定是否通知，例如：

    • GDPR：若個資外洩，72 小時內通知監管機關
    • ISO 27001：依照 事件應變計畫（IRP） 處理通報

    若尚未確認攻擊範圍，貿然通知客戶，可能引發不必要的恐慌或法律問題。

結論：

當系統遭受攻擊時，第一步應該是 (C) 確認並評估攻擊的範圍和影響，然後再進行 遏制（Containment）、修復（Eradication & Recovery）、通報（Notification）。

    • 避免風險（不執行高風險操作）

    • 資訊分類與存取控制政策
    • 風險管理原則

  • ✅ 風險評估後 → 必須先有「風險處理框架」，才能開始施行對策！
  • ✅ ISO 27001 是風險導向的框架，所有安全控制措施都應該基於風險評估的結果來決定！

  • 風險管理是核心，不能跳步！

(A) 組織的業務需求和目標

解析：

在實施資訊安全防護機制時，最重要的是確保 安全措施符合組織的業務需求與目標，而不是單純增加安全性。資訊安全應該是「業務的支撐」，而不是業務的阻礙！

在 ISO 27001 的 風險管理與安全控制 過程中，以下因素是關鍵考量點：

1️⃣ 業務需求 → 企業運作方式、安全需求（如金融業 vs. 軟體業的需求不同）

2️⃣ 合規要求 → 符合法規（如 GDPR、ISO 27001、NIST CSF）

3️⃣ 風險管理 → 適當的資安風險控制措施

因此，在實施資訊安全防護機制時，應確保其不影響 業務運營，並與組織目標一致。

選項分析：

  • ✅ (A) 組織的業務需求和目標 → 正確！

    業務需求驅動資安策略：如果安全機制影響業務運行，員工可能會繞過安全措施，反而降低整體安全性。

    範例：

    • 銀行的 ATM 需要嚴格身份驗證，但也要確保交易流程流暢，不能過度影響使用者體驗。
    • 軟體開發公司需要保護原始碼，但開發人員仍需順暢地存取版本控制系統（如 Git）。
  • ❌ (B) 組織的市場競爭優勢 → 錯誤！

    市場競爭優勢 主要與產品、商業模式、定價策略有關，而不是資訊安全的直接考量點。

    儘管資安做得好可能成為競爭優勢（如 Apple 強調隱私安全），但市場競爭優勢並不是設計資安機制的主要因素。

  • ❌ (C) 組織的企業形象 → 錯誤！

    企業形象 雖然與資安有關（例如資料外洩會損害品牌信譽），但它應該是 資安策略的影響結果，而非決策依據。

    範例：

    • Facebook 在 2018 年的 Cambridge Analytica 事件損害企業形象，但當初的資安決策並不是基於企業形象，而是業務需求。
    • 許多公司選擇合規（如 GDPR），部分原因是避免公關災難，但首要考量仍然是法規與業務需求。
  • ❌ (D) 組織的投資組合 → 錯誤！

    投資組合（Investment Portfolio） 與資安策略無直接關聯，這通常是財務部門考量的問題，而不是資安管理的核心因素。

    無論企業的投資決策如何，資訊安全仍需按照風險管理原則來執行。

結論：

📌 資訊安全策略應該與組織的業務需求和目標一致，避免影響業務運營，並在風險可接受範圍內實施控制措施。

📌 考試技巧：

  • 遇到 ISO 27001 或 資安管理決策 相關題目，先思考 業務需求 vs. 資安風險。
  • 「資訊安全是業務的支撐，而非阻礙」 是核心概念！

RTO 指的是 業務流程中斷後，必須在多長時間內恢復到可接受的運作狀態，這對災難復原（DR）計畫至關重要。

🔍 解析

1. syslog 伺服器使用標準埠號 514（UDP 預設，TCP 可選）

2. ping 測試成功，但 syslog 傳輸失敗

  • ICMP（ping）與 UDP/TCP 屬於不同的協議。
  • ping 成功只代表網路連線正常，不代表 syslog 流量可以通過。
  • 防火牆可能只允許 ICMP，而封鎖了 UDP 514！

3. 解決方案

  • 最合適的方式是 在防火牆上放行 UDP 514 流量，讓 syslog 傳輸正常運作。
  • 如果 syslog 伺服器使用 TCP 514，則需放行 TCP 514。

  • ❌ (A) 重新安裝伺服器 A 的 syslog → 錯誤

    伺服器 B 可以 ping 到 A，代表伺服器 A 本身沒有問題。

    問題應該是防火牆阻擋 UDP 514，而不是 syslog 服務有誤。

  • ❌ (C) 移除保護伺服器 A 的防火牆設備 → 錯誤

    這會降低伺服器 A 的安全性，暴露在潛在攻擊中！

    適當的做法是「調整防火牆規則」，而不是移除防火牆。

  • ❌ (D) 將伺服器 B 移至與伺服器 A 相同的網段 → 錯誤

    syslog 傳輸與網段位置無關，而是受防火牆規則影響。

    就算兩台伺服器在相同網段，如果防火牆仍然阻擋 UDP 514，問題依舊存在！

✅ 最佳解法

    若 syslog 使用 TCP，則需開放 TCP 514。

(A) 使攻擊者可以讀取到機密資訊 ✅ （正確）

(B) 使攻擊者可以關閉系統 ❌ （錯誤）
  • LFI 本身只能讀取伺服器上的文件，不會影響系統的運行或關閉系統。
  • 若攻擊者能讀取高權限配置檔（如 /etc/shadow 或 config.php），進一步利用其他漏洞（如 RCE），才有可能影響系統。

(C) 使攻擊者可以獲得系統管理員的權限 ❌ （錯誤）
  • LFI 本身不會直接讓攻擊者獲取 root 權限。
  • 但如果攻擊者讀取到某些機密檔案（如 SSH 私鑰、密碼文件等），再搭配其他漏洞，可能會提升權限。
  • 但 LFI 只是提供讀取權限，不會直接讓攻擊者變成系統管理員，所以這個選項並不正確。
(D) 使攻擊者可以對網路流量進行監控 ❌ （錯誤）
  • LFI 影響的是伺服器本地文件的存取，與網路流量監控無關。
  • 若攻擊者讀取到 tcpdump 或 wireshark 之類的封包記錄檔，可能會間接獲取部分封包內容，但這仍然與「監控網路流量」不同。
  • 真正的流量監控通常需要 ARP 欺騙、網路釣魚、中間人攻擊（MITM）等手段，LFI 無法達成此目的。

總結

✅ 本地文件讀取漏洞（LFI）最主要的風險是「機密資訊洩露」，因此正確答案為 **(A)**。

    • 正確：保持 Windows 主機和 WSL2 子系統的軟體版本更新是最基本且有效的安全措施，能修補已知的安全漏洞並增強系統防護。

    • 錯誤：禁用防火牆和安全軟體會讓系統暴露在攻擊中，尤其是開放的網路服務、共享資源等。

    • 錯誤：共用同一組系統管理員密碼會增加風險，若其中一個系統被攻擊者入侵，可能導致整個開發環境失守。
    • 正確做法：應使用不同的管理帳號與密碼，並實施多因素驗證來增強安全性。

最佳實務建議：

  • 定期更新：保持 Windows 主機與 WSL2 子系統的最新狀態，避免已知漏洞被攻擊者利用。

  • 啟用防火牆與安全軟體：設定Windows 防火牆與安全工具，監控並保護 WSL2 的網路活動。
  • 隔離與存取控制：確保 Windows 和 Linux 子系統之間的存取權限和共享資源受到控制，避免不必要的風險。
  • 不同的帳號與密碼：為 Windows 和 WSL2 使用不同的管理密碼，並建議使用多因素驗證。

  • 使用 網路監控工具來分析 WSL 內的網路行為。

    • 分析： 網路流量清洗服務（或稱 DDoS 緩解服務）是專為應對大規模 DDoS 攻擊而設計的。它們能將受攻擊目標的流量引導至專用的清洗中心，在那裡識別並過濾掉惡意流量，只將合法流量轉發給目標。對於混合式 DDoS 攻擊，特別是高容量攻擊，這是目前最有效且推薦的解決方案之一。
    • 判斷： 非常合適。

    B) 可透過限制 DNS 查詢次數，來緩解 DDoS 攻擊

    • 分析： 限制 DNS 查詢次數（Rate Limiting）是針對 DNS 伺服器應用層攻擊的有效防禦措施。它可以防止單一來源或有限的惡意來源透過發送大量查詢來耗盡 DNS 伺服器的處理資源。然而，面對混合式 DDoS 攻擊，尤其是當攻擊流量來自高度分散的殭屍網路，且每個惡意來源的查詢速率都保持在限制以下時，僅靠此措施可能無法完全阻止聚合流量對頻寬或伺服器整體資源的消耗。雖然這是一項重要的安全配置，但在大規模分佈式攻擊面前，其獨立的緩解能力有限。
    • 判斷： 不夠合適（因其在應對高度分散和大規模攻擊時的局限性）。

    • 分析： 這項敘述的前提「如果網路頻寬無須考量」從根本上與 DDoS 攻擊的定義相悖。DDoS 攻擊的關鍵目標就是透過巨量流量耗盡目標的網路頻寬，使其無法提供正常服務。如果頻寬無限，那麼流量型的 DDoS 攻擊本身就無法達成其主要目的。在實際情況中，頻寬總是一個重要的考量因素。雖然過濾機制（例如防火牆規則、ACL）是 DDoS 防護的組成部分，但在缺乏足夠頻寬的情況下，這些本地過濾措施在流量抵達網路設備並被處理之前，可能就已經導致上游鏈路飽和。因此，將此措施建立在一個不切實際的假設上，使其成為不適用於實際 DDoS 情境的建議。

    • 判斷： 非常合適。

    由於 CSRF 攻擊依賴於構造一個預先確定的惡意請求，如果每次登入時 CSRF Token 都會更新，那麼攻擊者就無法在不知道當前有效 Token 的情況下構造出一個能夠通過伺服器驗證的惡意請求。即使攻擊者誘騙用戶點擊了惡意的鏈接或表單，由於其中缺乏正確且最新的 CSRF Token，伺服器也會拒絕該請求，從而防止了 CSRF 攻擊的成功。

    其他選項：

    • (A) SQL 注入（SQL Injection）： SQL 注入是通過在輸入字段中插入惡意的 SQL 語句來攻擊資料庫。登入錯誤訊息和 CSRF 防護與 SQL 注入的防護機制無關。
    • (C) 指令注入（Command Injection）：： 指令注入是通過在輸入字段中插入惡意的操作系統命令來攻擊伺服器。登入錯誤訊息和 CSRF 防護與指令注入的防護機制無關。
    • (D) 跨站指令碼（Cross Site Scripting，XSS）： XSS 是通過在網站中注入惡意的 JavaScript 或其他客戶端腳本，使其在其他用戶的瀏覽器中執行。登入錯誤訊息和 CSRF 防護與 XSS 的防護機制無關。
